## §A — Economic-Security Attack Scenarios

This section runs specific attack stories end-to-end. Each scenario lists the attacker, the assumed protocol parameters, the attack steps, the protocol's loss, and the mitigation that closes the attack (or downgrades it to acceptable).

### A.1 — Wash-trade reputation farming → high-value abscond

**Attacker**: One human controlling ≥3 wallets (one assistant, ≥2 "users").
**Assumption**: No identity primitive (ADV-F-009 still open). Collateral required = $X. Reputation tier rewards include access to higher-value orders, fee rebates, bonus pool eligibility.
**Steps**:
1. Attacker opens assistant account with $X collateral.
2. Attacker opens N user accounts, each funded just enough to place small orders.
3. Attacker self-orders N times. Each completes optimistically (no real dispute). Reputation accrues.
4. Once attacker reaches high tier, attacker accepts one large genuine order of value $Y, with $Y > $X.
5. Attacker absconds: takes payment, never delivers, exits before dispute window expires (or wins dispute via Sybil-controlled juror selection if juror selection is reputation-weighted).
**Protocol loss**: $Y − $X (collateral slash, if slashing works) + insurance pool drain to make user whole + reputation damage.
**Mitigation that closes**: $X scaled with reputation tier — i.e., `collateral_required = max($X_base, k * worst_recent_order_value)`. Sybil cost grows with target. Combined with proof-of-personhood at sign-up, attack becomes uneconomic.

**Owner**: Tokenomics + Security Lead. Specify `k` and decay parameters before reputation tiers go live.

### A.2 — Optimistic-release griefing (silent-accept exploitation)

**Attacker**: Assistant with low integrity or organized assistant ring.
**Assumption**: Optimistic release with challenge window `T`. User is the only one who can affirmatively reject. If user is silent at `T+1`, escrow auto-releases.
**Steps**:
1. Attacker takes a real order. Delivers partial / defective work.
2. User opens the order, doesn't notice the defect until later. Window expires.
3. Escrow auto-releases full payment to attacker.
4. Attacker accumulates these incidents across many users (each individually small, no incentive to dispute).
**Protocol loss**: Many small uncovered consumer harms. Cumulatively: reputation, regulatory, and class-action exposure.
**Mitigation that closes**:
- Require **affirmative user acceptance** for any release > $threshold. Below the threshold, silent acceptance is OK because the harm-per-incident is bounded.
- Reputation-weighted threshold (newer assistants need affirmative accept on more orders).
- Cooling-off period (e.g., 72h after optimistic release, user can still file a dispute that re-locks funds from assistant's collateral).
- Active outbound prompt to user at `T-24h`.

**Owner**: Lead SC Engineer (state machine in escrow program, ADV-F-003).

### A.3 — Dispute-system bribery / collusion

**Attacker**: Litigant in a dispute, with budget $B.
**Assumption**: On-chain jury, N jurors per dispute, jury reward $R, slashing on jury misbehavior limited or zero.
**Steps**:
1. Attacker identifies jurors as soon as they're selected (if selection is public).
2. Bribes ⌈N/2⌉ jurors, offering $B/N each. If `$B/N > expected_loss_from_slash_or_reputation`, jurors accept.
3. Jury rules in attacker's favor.
4. Attacker wins disputed funds; pays bribes; net positive iff `disputed_amount - $B > 0`.
**Protocol loss**: Disputed amount + protocol legitimacy.
**Mitigation that closes**:
- Hidden juror identity until reveal (commit-reveal voting; pre-vote, even jurors don't know who else is on the jury).
- Slashing of jurors found to vote against revealed-evidence consensus (appeal mechanism with a wider jury or appellate authority).
- Jury reward > expected bribe amount in median case.
- Cap on disputed amount per jury size (large disputes route to larger juries automatically).
- Optimistic dispute: assume initial juror is honest, allow challenges that escalate. Limits the parties that need to be bribed.

**Owner**: Lead SC Engineer + Security Lead (dispute spec, ADV-F-004).

### A.4 — Front-running listing acceptance (MEV)

**Attacker**: Validator/relayer or an assistant with infrastructure access.
**Assumption**: High-value listing has constrained eligibility but anyone meeting it can accept. Multiple assistants try to accept simultaneously.
**Steps**:
1. Attacker monitors mempool / order flow for new high-value listings.
2. Submits accept-tx with higher priority fee or via Jito bundle, beating other honest assistants.
3. Repeat. Honest assistants drop off the platform.
**Protocol loss**: Assistant churn. Reputation system becomes meaningless (whoever has the best infra wins).
**Mitigation that closes**:
- Commit-reveal acceptance: assistants commit a sealed accept within window `T1`; reveal in `T2`. Listing creator selects from reveals using deterministic rule (highest reputation, or random among top-K).
- Or: dutch-auction style: listing creator sets max price; reservation declines over time; assistants accept when they're satisfied. Removes the race entirely.
- Or: explicit listing-creator approval before escrow funds (listing creator picks among multiple offers).

**Owner**: Lead SC Engineer (marketplace state machine).

### A.5 — Stablecoin freeze-induced denial-of-service

**Attacker**: External — regulator or court — freezes a USDC PDA.
**Assumption**: Single-stable (USDC) escrow.
**Steps**:
1. OFAC / court orders Circle to freeze BlockAssist escrow PDA or a specific user's funds.
2. All transactions touching that PDA fail. Escrow becomes stuck.
**Protocol loss**: Locked funds; protocol degrades to read-only for affected accounts.
**Mitigation that closes**:
- Add a documented `escrow_frozen` terminal state (graceful degradation, not transaction revert).
- Per-escrow PDA design (one PDA per order) so a freeze isolates to one order, not the whole protocol.
- Document this risk transparently in ToS.
- Long-term: evaluate dual-stable (PYUSD + USDC) but that doubles complexity.

**Owner**: Lead SC Engineer + Legal (when LEGAL resumes).

### A.6 — Insurance-pool drain via coordinated false claims

**Attacker**: Coordinated ring of users + assistants.
**Assumption**: Insurance pool covers user losses on assistant default. Claim review is jury-based.
**Steps**:
1. Ring opens many small orders that intentionally fail in ways that trigger insurance payouts.
2. Each individual claim looks legitimate (no admin red-flag).
3. Cumulative payouts > pool reserves.
**Protocol loss**: Pool insolvent → uncovered losses on next genuine claim → user trust collapse.
**Mitigation that closes**:
- Per-user/per-assistant payout cap.
- Aggregate cap per epoch with rollover queue.
- Statistical anomaly monitoring (claim rate vs baseline) with circuit breaker.
- Insurance pool reserve ratio target ≥ 4× rolling 30-day claim volume; pause new high-value orders if breached.

**Owner**: Tokenomics + Security Lead (gated on ADV-F-007 vehicle).

### A.7 — Marketplace listing spam DoS

**Attacker**: Anyone with a small budget who wants to degrade the platform.
**Assumption**: Free listing creation.
**Steps**:
1. Attacker scripts 10,000 listing creations per hour.
2. UI search degrades. Honest listings drowned.
**Protocol loss**: UX, possibly RPC costs.
**Mitigation that closes**:
- Listing fee (refundable on completion, forfeit on cancel) — small enough to not block honest creators, large enough to make 10k/hr expensive.
- Per-account rate-limit at the indexer layer (not on chain — on-chain rate-limiting wastes compute).
- Off-chain listing UI filter using reputation.

**Owner**: Lead SC Engineer (listing fee) + Mobile/Web Leads (UI filter).

### A.8 — Privy session theft via passkey side-channel

**Attacker**: Holds the victim's unlocked device for a short window.
**Assumption**: Passkey unlocks the Privy session. If device is unlocked, passkey is silently usable.
**Steps**:
1. Attacker borrows victim's unlocked phone for 30s.
2. Opens BlockAssist app. Passkey authenticates without re-prompting.
3. Transfers max allowed amount to attacker-controlled address.
**Protocol loss**: Per-incident bounded by transfer limit; aggregate trust damage if it happens publicly.
**Mitigation that closes**:
- Per-tx biometric re-confirmation for high-value sends (not just session auth).
- Velocity limits on outbound sends.
- Time-locked recovery if attacker tries to add a new device or change the destination.

**Owner**: Mobile Lead.

### A.9 — Reorg / commit hash race

**Attacker**: Validator with control of recent block production (rare on Solana but possible during halts).
**Assumption**: Standard Solana finality assumptions.
**Steps**:
1. Order acceptance tx submitted; appears finalized.
2. Cluster halts and resumes from a slot pre-acceptance.
3. Conflicting tx replaces it.
**Protocol loss**: Order acceptance race; one party thinks they have the order, the other doesn't.
**Mitigation that closes**:
- Wait for `confirmed` (or `finalized` for high-value) before showing acceptance to either party.
- Idempotent acceptance: client retries the accept tx on UI side until on-chain status agrees.
- Don't ship UI states that imply finality before consensus does.

**Owner**: Mobile/Web Leads + SDK.

### A.10 — Replay across devnet/mainnet

**Attacker**: Anyone observing a signed tx on devnet.
**Assumption**: Recent blockhashes are not strictly distinct across clusters; signatures could replay if program logic doesn't bind to cluster.
**Steps**:
1. Devnet test reveals user-signed transaction.
2. Attacker attempts replay on mainnet.
**Protocol loss**: Usually zero, because Solana's recent blockhash is cluster-local. But program logic that signs over relative state (e.g., a Privy off-chain meta-tx) may be replayable.
**Mitigation that closes**:
- Always include cluster-specific nonce or sysvar.cluster.
- Treat meta-tx signatures as one-time; persist a nonce/replay table per user.

**Owner**: Lead SC Engineer (defensive coding).

## §B — Dispute-System Landscape Verdict

**Question**: Is there a production-grade on-chain jury primitive on Solana we can reuse, or do we build?

### B.1 — What we surveyed

(Reviewer's best-effort scan as of 2026-05; Lead SC Engineer must verify currency.)

| Candidate | Status | Fit |
|---|---|---|
| **Realms / SPL Governance** | Production. Token-weighted voting on proposals. | Wrong shape — designed for protocol governance, not per-dispute juries. Adapting it forces token-weighted juries (whale capture). |
| **Squads multisig** | Production. m-of-n signers. | Wrong shape — fixed signer set, not random juror selection. |
| **Switchboard VRF + custom logic** | Production VRF. | A *primitive*, not a jury system. Useful for juror selection within a jury system we build. |
| **UMA Optimistic Oracle (Solana port?)** | Rumored, not verified. | Best fit conceptually (optimistic + escalation), but Solana port maturity unverified. |
| **Kleros (cross-chain via bridge)** | Cross-chain. | v2.1 explicitly demoted this to experimental (bridge risk). |
| **Reputation-weighted custom jury** | New build. | Forces us to ship the Sybil primitive (ADV-F-009) first. |

### B.2 — Reviewer's verdict

**Build, but optimistically.** The honest assessment:

1. There is no off-the-shelf, audited, production-on-mainnet Solana jury primitive that ships out of the box.
2. The cheapest path is an **optimistic dispute** model where:
   - 90%+ of disputes settle at level 0 (initial mediator, single human in the team, fast turnaround).
   - The minority that escalate go to a jury — which we build, using Switchboard VRF for selection and commit-reveal voting.
   - We *defer* the appellate / cross-jury layer until volume justifies it.
3. The "Kleros experimental" demotion in v2.1 was correct, but the deferred work didn't get replaced with a concrete plan. This deep dive proposes the replacement.

### B.3 — Phased plan

**Phase 1 (MVP, single pilot city)**: Level-0 mediator only. A named role inside the protocol team, with publicly disclosed acceptance/rejection rate, conflict-of-interest disclosure, and an SLA. Cheap, fast, legally simple. Trade-off: not "decentralized" but appropriate for the volume.

**Phase 2 (2nd city, more volume)**: Add level-1 jury for escalations. 5–9 jurors selected by Switchboard VRF from a pool that has staked collateral. Commit-reveal voting. Bribery mitigation per §A.3.

**Phase 3 (post-MVP)**: Appellate layer (larger jury or DAO vote) for disputes above $threshold or after specific number of appeals.

**Recommendation**: Update `docs/architecture/v2-architecture.md` (currently 11 lines, very thin) with this phased plan. Lift the "Solana-native on-chain jury fallback" tagline from v2.1's blanket statement into a concrete level-by-level spec.

**Owner**: Lead SC Engineer to draft `docs/architecture/dispute-system.md` in next sprint.

## §D — Cross-Platform Mobile Implementation Risks

The mobile app is React Native + Expo + Privy passkeys. Cross-platform sounds simple; in practice the iOS/Android divergence around passkeys, secure enclaves, and app-store policy is significant.

### D.1 — Passkey divergence

**iOS (Apple)**:
- Passkeys backed by iCloud Keychain. Sync across devices automatically (which can be a privacy concern but a UX win).
- AutoFill UI is system-provided; cannot be styled. Users see a system sheet.
- Requires `webcredentials` association file (`apple-app-site-association`) on the protocol's domain. Misconfigured → silent failure.
- Privacy: passkey credential IDs are visible to Apple, not the relying party.

**Android (Google)**:
- Passkeys backed by Google Password Manager *or* the user's chosen credential manager (Samsung Pass, 1Password, etc.). Less uniform.
- AutoFill UI varies by manager.
- Requires `assetlinks.json` on the protocol's domain.
- Older Android (< 14) has limited passkey support; many users still on Android 11/12. **This is a real exclusion risk.**

**Risks**:
- A working iOS implementation does not imply working Android. Test both early and often.
- Different recovery flows per platform → must document for support team.
- Passkey-not-supported fallback (older Android, custom OEMs) needs a path — magic-link email? Hardware key?

**Mitigation**:
- Privy SDK abstracts much of this but version-pinning matters (mobile/package.json currently has `^0.XX.X` placeholder — must be replaced).
- Detect platform and feature-flag accordingly; never assume passkey works.

### D.2 — Expo router gotchas

The current mobile app uses Expo Router (file-based routing in `app/`).

**Known pain points**:
- Deep links across iOS Universal Links and Android App Links require separate configuration and verification.
- Web preview (Expo for Web) does not have the same routing semantics as native; if web is in scope, set expectations.
- Privy `@privy-io/expo` requires native module config — not just JS. Means EAS Build or a custom dev client, not standard Expo Go.

**Mitigation**:
- Document the EAS Build workflow in `docs/architecture/mobile-build.md` (create).
- Pin Expo SDK version; upgrade only on scheduled cadence.

### D.3 — App-store policy traps

**Apple App Store**:
- Apps that broker financial transactions outside the platform face increased scrutiny.
- Crypto wallets are allowed but reviewed by a specialized team; expect 1–2 week review for first submission and after major changes.
- In-app token sales / staking interfaces are commonly rejected unless tied to "approved cryptocurrency exchanges" — relevant if $ASSIST staking UI appears in app.
- Required: full KYC flow visible to App Review reviewers.

**Google Play**:
- Crypto allowed but DeFi staking specifically faces scrutiny.
- Required: Play Console "Personal & Sensitive Data" declaration covering wallet/finance categories.

**Risks**:
- A token-staking UI in the app may trigger rejection or removal post-launch.
- App removal mid-pilot is a worst-case scenario; we need a contingency (PWA fallback?).

**Mitigation**:
- For the no-token track: app contains no staking UI, no token sale, just escrow + reputation + payments. Lower review risk.
- For the token track: keep staking out of the mobile app entirely; expose only via web. Reduces store risk.
- Submit to both stores early with a minimum-feature build to surface review concerns before they block the launch.

### D.4 — Push notifications and order timing

The escrow optimistic-release model (§A.2 mitigation) requires reliable outbound notifications at `T-24h`. APNs/FCM are reliable but not guaranteed. If a user misses the notification and the window expires, the protocol failed them.

**Mitigation**:
- Don't bind release decisions to push-notification delivery. Use silent in-app indicators *and* email *and* SMS as fallback.
- Long challenge windows (≥72h) tolerate intermittent notification delivery better than short ones.

### D.5 — Recommendation

Create `docs/architecture/mobile-platform-risks.md` capturing items §D.1–D.4 with platform-specific runbooks. Owner: Mobile Lead. Sprint 1 deliverable.
